# JD-Mapping Methodology

## What to extract from a JD

For each JD, identify and list:

### 1. Repeated phrases (highest signal)

Words/phrases used multiple times in the JD are the recruiter's vocabulary. Weave these naturally into the resume where they map to real work. **Don't keyword-stuff.**

Example patterns to watch for:
- "force multiplier" → surface in leadership skills cell
- "fault-tolerant" → surface in relevant architecture bullets
- "distributed systems" → surface in summary opener
- "cloud-native architectures" → surface in cloud skills cell

### 2. Hard requirements

The "must have" list. Each item gets a corresponding signal in the resume — or an honest gap if the candidate doesn't have it.

### 3. Highly preferred items

Discriminators between candidates with similar core fit. If the candidate has these, surface prominently. If not, decide whether to address proactively (cover letter, interview prep) or stay silent.

### 4. Listed technologies

Map each to either:
- ✓ Real hands-on work → list and surface in bullets
- ~ Evaluation/PoC level → list with care, prepare honest framing for interview
- ✗ Don't have → leave off entirely, don't fabricate

### 5. Compensation band (if listed)

Signals the seniority bar of the interview:
- **Entry/mid band**: standard technical screen, moderate depth expected
- **Senior band**: rigorous technical and some leadership questions
- **Principal/Staff/Distinguished band**: every claim cross-examined, system design at scale, expect 6+ interview rounds

### 6. Company context cues

- "Embedded in AWS/GCP/Azure" → multi-cloud experience matters
- "Open-source X, Y, Z" → community/contributor signal valued
- "Hybrid" → they care about in-office presence

## Where to put each signal

| Signal type | Resume location |
|---|---|
| Title/scope of role | Summary opener + headline subtitle |
| Repeated JD phrases | Naturally woven into Summary + Skills + bullets |
| Hard-requirement tech | Skills section + relevant experience bullets |
| Leadership patterns | Leadership skills entry + verb choices in bullets |
| Specific accomplishments | Bullets with quantified outcomes |
| Publications/IP | Dedicated bullet — use accurate terminology |

## Things NOT to do

- **Don't write a summary that doesn't match the bullets.** If the summary says "deep Kafka experience" and no bullet supports it, the contradiction signals fabrication.
- **Don't sprinkle JD keywords without context.** A term on a skills list when the candidate hasn't worked with it is a trap — the first interview question will expose it.
- **Don't rewrite work history to match a JD.** Each role's bullets describe what the candidate actually did. Customization happens in *which* bullets are included and *how prominently* they're framed.
- **Don't move bullets between roles to align with JD chronology.** Each bullet belongs in the role where the work happened.

## Title strategy decision matrix

| JD title | Typical match guidance |
|---|---|
| Distinguished / Fellow | If candidate holds that title: match. If one level below: use actual title, frame scope and impact to show near-level readiness |
| VP / SVP | Principal or Senior Architect level, depending on actual scope — don't claim VP if never held it |
| Staff Engineer | Principal Engineer (one notch up from Senior) is usually defensible if the candidate has principal-level scope |
| Senior Principal / Principal | Direct match if candidate holds the title |
| Senior Engineer / Architect | Match if appropriate; consider whether candidate is applying to step up |
| Solutions / Cloud Architect | Match with specific domain qualifier from the JD |

Always confirm title strategy with the candidate before locking in — referral angle, company culture, and personal positioning goals affect the call.
